For a long time I wrote Python the way most people start: no type annotations, just variables flying around and hoping the function receives what you think it does. It worked, until it didn't. What actually changed my mind was a class where my professor refactored a messy untyped function live, and suddenly the editor was catching bugs before we even ran anything.

## The Problem With Untyped Code

Here is the kind of code I used to write without thinking twice about it:

```python
def get_discount(price, discount):
    return price - (price * discount)

total = get_discount("19.99", 0.1)
```

This runs without error. Python happily tries to multiply a string by a float and then crashes at runtime with a `TypeError`. You only find out something is wrong when the code actually executes, which in a real app might be deep inside a user flow.

## Adding Type Hints

Python introduced type hints in [PEP 484](https://peps.python.org/pep-0484/) (Python 3.5). They don't change how the code runs, but they tell you and your tools what types are expected:

```python
def get_discount(price: float, discount: float) -> float:
    return price - (price * discount)

total = get_discount("19.99", 0.1)  # editor flags this immediately
```

Now your editor (and tools like [ty](https://github.com/astral-sh/ty)) will underline that string argument in red before you run a single line. The bug moves from runtime to edit time.

## It Gets More Useful With Complex Data

Where type hints really earn their place is when you're passing structured data around:

```python
# without types — you have no idea what this function expects
def process_user(user):
    return user["name"].upper()

# with types — the contract is explicit
def process_user(user: dict[str, str]) -> str:
    return user["name"].upper()
```

Even better, you can use `TypedDict` or dataclasses to define the shape of your data properly:

```python
from typing import TypedDict

class User(TypedDict):
    name: str
    age: int

def process_user(user: User) -> str:
    return user["name"].upper()

process_user({"name": "Simon", "age": "not a number"})
# ^ ty catches this: 'str' is not assignable to 'int'
```

## What You Get for Free

Once you add types, your editor can autocomplete fields on objects, warn you about impossible operations, and navigate your codebase more accurately. Refactoring becomes much safer because the type checker tells you everywhere a change breaks something, rather than you finding out through tests or worse, production.

## Conclusion

You don't have to annotate everything at once. Python's type system is gradual, meaning you can add hints file by file and the untyped parts just get ignored. Start with function signatures, especially ones shared between modules, and the value shows up quickly. My professor's point was simple: the earlier you catch a bug, the cheaper it is to fix. Types are one of the cheapest ways to catch them earlier.

I remember very clearly the last sentence that made me switch to typed Python: "When you run your code you spend hours debugging types, I just spend 5 more minutes thinking about it and writing it down while coding".
